# ShopCopilot AI 项目进化史与技术总结

## 一、项目背景

**项目名称**：ShopCopilot - 店铺智能助手
**项目定位**：面向 DIY 手工店、亲子游乐等体验式门店的 AI 智能助手
**技术栈**：FastAPI + LangChain + LangGraph + MiMo LLM + ChromaDB + MySQL + Redis

---

## 二、进化历程

### Phase 1：毛坯阶段（v1.0）

**架构**：简单的 RAG + NL2SQL

```
用户问题 → NL2SQL 查询 → 返回结果
```

**技术点**：
- 基础 RAG 检索
- 简单的 NL2SQL
- 无工具调用能力

**问题**：
- 无法执行操作（退款、发券等）
- 无法理解上下文
- 无法处理复杂问题

---

### Phase 2：工具化阶段（v2.0）

**架构**：引入工具调用

```
用户问题 → Router → 工具调用 → 返回结果
```

**技术点**：
- 定义 9 个操作工具（退款、发券、核销等）
- 工具通过 Java 后端 API 执行
- 引入 LangChain Tool Calling

**创新点**：
- 工具与 Java 后端解耦（AI 服务只负责理解，Java 负责执行）
- 所有操作工具统一调用 Java API

---

### Phase 3：Agent Loop 阶段（v3.0）

**架构**：引入 ReAct 循环

```
用户问题 → LLM 决策 → 调用工具 → 观察结果 → 继续决策 → 最终回答
```

**技术点**：
- LangGraph 实现 ReAct 循环
- LLM 自主决定调用哪些工具
- 支持多轮迭代

**创新点**：
- LLM 可以自主探索和验证信息
- 不是死板的单次调用，而是迭代推理

---

### Phase 4：智能路由阶段（v4.0）

**架构**：引入 Router 和 Supervisor

```
用户问题 → Router → Agent 选择 → 执行 → 返回结果
                ↓
        Supervisor（多任务拆分）
```

**技术点**：
- `_check_question` 一次 LLM 调用完成 5 个判断
- COMPLEXITY_PROMPT 智能路由
- Supervisor 多任务拆分和执行

**创新点**：
- 统一的路由判断（有效性、上下文、追问、操作、查询）
- 确认性回复智能理解（"是的" → 根据上下文判断意图）

---

### Phase 5：批量操作阶段（v5.0）

**架构**：支持多任务批量确认

```
用户: "批准黄晓明，拒绝赵丽颖"
    ↓
Router → multi 模式
    ↓
Supervisor 拆分:
  子任务1: NL2SQL 查询退款 ID
  子任务2: refund_approve(黄晓明)
  子任务3: refund_reject(赵丽颖)
    ↓
batch_confirm → 用户确认 → 批量执行
```

**技术点**：
- SubTask 增加 `tool_name` 字段
- Supervisor 支持操作型子任务的确认流程
- `POST /api/chat/batch_confirm` 批量执行
- 前端 BatchConfirmCard 组件

**创新点**：
- 查询任务和操作任务混合执行
- 查询结果自动传递给后续操作（query_context）
- 批量确认弹窗支持 checkbox + 输入框

---

### Phase 6：自主探索阶段（v6.0，当前）

**架构**：LLM 具备自主探索能力

```
用户问题 → Agent Loop
    ├── execute_sql_query（通用 SQL）
    ├── list_tables（列出数据库表）
    ├── describe_table（查看表结构）
    ├── search_docs（搜索知识库）
    └── 探索提示词（鼓励主动探索）
```

**技术点**：
- 新增 3 个探索工具
- 系统提示词增加探索策略
- NL2SQL 输出人类可读格式
- 后台系统导航能力

**创新点**：
- LLM 遇到不确定信息时主动查询验证
- 不猜测，不编造，查询验证
- 无法解决时引导用户到后台系统

## 三、技术架构总结

### 核心组件

| 组件 | 技术 | 作用 |
|------|------|------|
| **Router** | LLM + COMPLEXITY_PROMPT | 智能路由（查询/操作/知识/分析） |
| **Agent Loop** | LangGraph ReAct | LLM 自主决策和工具调用 |
| **Supervisor** | 任务拆分 + 批量执行 | 多任务编排和确认 |
| **NL2SQL** | 并行候选 + 自动修复 | 自然语言转 SQL |
| **RAG** | ChromaDB + CRAG + Self-RAG | 知识检索和问答 |
| **工具体系** | LangChain Tools + Java API | 9 个操作工具 + 探索工具 |

### 数据流

```
用户输入 → Router → Agent 选择
    ↓
Agent Loop（ReAct 循环）
    ├── NL2SQL: 查询数据库
    ├── 操作工具: 调用 Java API
    ├── 探索工具: 查询表结构/知识库
    └── RAG: 检索知识库
    ↓
结果汇总 → 流式输出 → 前端渲染
```

## 四、创新点

### 1. 统一的路由判断

**传统方式**：硬编码规则判断路由
**创新方式**：LLM 一次调用完成 5 个判断（有效性、上下文、追问、操作、查询）

```python
# 传统：多个 if/else
if "查询" in message:
    return nl2sql
elif "退款" in message:
    return tool

# 创新：LLM 综合判断
check_result = await _check_question(task, shop_name, history_context)
# 返回: is_valid, is_context_question, need_clarify, is_operation, need_requery
```

### 2. 确认性回复智能理解

**问题**：用户说"是的"，系统不知道确认什么
**创新**：根据助手上一个问题判断确认的内容

```
助手: "请问具体指哪两个的金额？"
用户: "是的"
→ LLM 判断：确认的内容是"查询金额" → need_requery=true
```

### 3. 批量确认机制

**问题**：多个操作需要逐个确认，体验差
**创新**：合并为一个批量确认弹窗

```
用户: "批准黄晓明，拒绝赵丽颖"
→ 拆分为 2 个操作
→ 合并为 1 个 batch_confirm 弹窗
→ 用户一次确认，批量执行
```

### 4. LLM 自主探索

**问题**：LLM 遇到不确定信息时放弃
**创新**：给 LLM 探索工具，鼓励主动查询

```
LLM 看到 status=1
→ 调用 execute_sql_query 查询 sys_dicts
→ 发现 1=处理中
→ 回答: "处理中的退款..."
```

### 5. 流式输出 + Session 一致性

**问题**：流式输出和 Session 保存内容不一致
**创新**：收集流式 chunks 保存到 Session

```python
# 收集流式输出
streaming_chunks = []
async for sse_event in _final_summarize_stream(...):
    yield sse_event
    streaming_chunks.append(chunk_content)

# 保存到 Session
self._ai_response_parts.append("".join(streaming_chunks))
```

## 五、经验教训

### 1. LLM 不是万能的

**教训**：LLM 有时会编造数据、误解指令
**解决**：
- 系统提示词明确禁止编造
- 给 LLM 探索工具验证信息
- 输出规范要求人类可读

### 2. 硬编码规则不可扩展

**教训**：硬编码的路由规则、参数解析规则难以维护
**解决**：
- 用 LLM 做判断（_check_question）
- 用 TOOL_REQUIREMENTS 声明参数需求
- 用 system_capabilities.json 描述后台能力

### 3. 上下文传递至关重要

**教训**：子任务之间不共享信息，导致重复查询
**解决**：
- query_context 传递查询结果
- history_context 传递对话历史
- 确认性回复基于上下文判断

### 4. 用户体验需要精心设计

**教训**：原始 SQL 输出、状态码数字对用户不友好
**解决**：
- CASE WHEN 映射状态码为中文
- 列名使用中文别名
- 流式输出 + 打字机效果

### 5. 安全不能妥协

**教训**：直接 SQL 操作数据库有风险
**解决**：
- 所有操作工具调用 Java 后端 API
- SQL 安全校验（注入防护 + shop_id 过滤）
- 只允许 SELECT 查询

## 六、面试要点

### 技术深度

1. **Agent 架构**：ReAct 循环 + LangGraph 状态图
2. **路由设计**：LLM 综合判断（5 个维度）
3. **批量操作**：任务拆分 + 确认流程 + 批量执行
4. **自主探索**：探索工具 + 系统提示词引导

### 创新点

1. **统一路由判断**：一次 LLM 调用完成 5 个判断
2. **确认性回复理解**：根据上下文判断"是的"确认什么
3. **批量确认机制**：多操作合并为一个确认弹窗
4. **LLM 自主探索**：遇到不确定信息主动查询验证

### 工程实践

1. **前后端分离**：AI 服务 + Vue3 管理后台 + UniApp 小程序
2. **Java 后端解耦**：AI 服务只负责理解，Java 负责执行
3. **流式输出**：SSE + 打字机效果 + Session 一致性
4. **安全机制**：Token 认证 + 角色权限 + SQL 安全

### 遇到的挑战

1. **LLM 输出不稳定**：JSON 解析失败 → 重试机制
2. **上下文丢失**：子任务之间不共享 → query_context
3. **状态码不可读**：原始数字 → CASE WHEN 映射
4. **批量操作复杂**：逐个确认 → 批量确认弹窗

### 未来改进

1. **Router 准确性**：持续优化 COMPLEXITY_PROMPT
2. **NL2SQL 复杂查询**：支持更复杂的 SQL 生成
3. **Agent Loop 效率**：限制探索次数，减少延迟
4. **前端 postMessage**：点击直接跳转后台页面
